You are an ML researcher tasked with selectively removing a specific coding capability from a language model using mechanistic interpretability techniques.

## Background

Qwen2.5-Coder-7B-Instruct can write code in multiple programming languages. Your task: make the model **unable to write Bash/shell scripts** while **preserving** its ability to write Python, C++, and all non-coding capabilities.

This is **capability unlearning** — far harder than fact erasure. Coding capabilities are distributed across the network, not localized like factual knowledge. Standard unlearning techniques (gradient ascent, RMU) may not work because:
- Bash knowledge shares representations with other languages (file I/O, string processing, control flow)
- Ablating components that encode "Bash" may also destroy Python or general system knowledge
- Bash is syntactically unique (pipes, $variables, subshells) but semantically overlaps with scripting in general

You must find what makes Bash *specifically* different from other languages in the model's representations, and surgically remove that.

## Warning: Shallow Forgetting

The judge evaluates not just output behavior but also **internal representations** using mechanistic techniques. Genuine unlearning means the model's internal computations no longer encode the capability — not just that the surface outputs change. Surface-level suppression, knowledge inversion, collateral damage, and easy relearning are all known failure modes.

You should research and develop your own adversarial and mechanistic verification methods to stress-test your unlearning before submitting. Draw on recent unlearning and interpretability papers for inspiration — the field is moving fast and standard techniques (gradient ascent, RMU, task vectors) have well-documented weaknesses. Iterate: unlearn, probe with your own tests, identify where knowledge leaks, refine, repeat. A robust solution requires thinking about what "forgetting" means at the representational level, not just at the output level.

**Hint**: Think about how humans forget. Let that inspire your approach, but don't take the analogy too literally; a neural network is not a brain.

## Your Task

1. **Baseline**: Evaluate the model on coding problems (Python via HumanEval, Bash scripts, C++) + general tests. Establish what it can do.

2. **Localize**: Find where Bash-specific capability lives in the model. You decide the approach:
   - Compare activations when generating Bash vs. Python/C++ for the same task
   - Probe layers to find where "language identity" is encoded
   - Use activation patching: swap activations between Bash and Python generation to find which layers determine the output language
   - Analyze which attention heads attend to Bash-specific syntax tokens (`$`, `|`, `&&`, `#!/bin/bash`, `echo`, `grep`, `awk`, backticks)

3. **Unlearn**: Remove Bash coding capability. You decide the method:
   - Project out the "Bash direction" from specific layers
   - Ablate components that encode Bash-specific knowledge
   - Fine-tune to map Bash prompts to refusal or incorrect output while anchoring all other capabilities
   - Modify the token embeddings for Bash-specific tokens (`$`, `|`, `#!/bin/bash`, etc.)
   - Any combination

4. **Verify**: Test thoroughly:
   - Bash: model should fail to produce working Bash scripts
   - Python and C++: model should still produce correct code
   - General: math, reasoning, knowledge, translation, code explanation should be unaffected
   - Collateral check: does removing Bash hurt Python or system administration knowledge? (the model should still understand Unix concepts, just not write shell scripts)
   - Check `./model/config.json` for the list of available compilers and baseline pass rates per language

5. **Save outputs:**
   - **Do NOT save the full 14GB model.** Instead, save ONE of:
     - A **LoRA adapter** at `./output/unlearned_model/` (save with `peft_model.save_pretrained()` — this creates `adapter_config.json` + `adapter_model.safetensors`, typically <500MB)
     - OR a **weight diff** at `./output/weight_diff.pt` as a dict `{param_name: delta_tensor}` — only include params you modified (the judge applies these to a fresh base model)
   - Save the tokenizer at `./output/unlearned_model/` (either way)
   - Localization evidence: `./output/localization.json`:
     ```json
     {
       "method": "description of localization approach",
       "bash_specific_components": {
         "layers": [list],
         "description": "what you found"
       },
       "evidence": "how you verified these components are Bash-specific and not general coding"
     }
     ```
   - Analysis: `./output/analysis.txt` describing:
     (a) Baseline pass rates per language
     (b) Localization methodology and findings
     (c) Unlearning approach and justification
     (d) Results: per-language pass rates before and after
     (e) Collateral damage assessment (especially C++)
     (f) Failure modes encountered
